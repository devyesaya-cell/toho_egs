# SYNC DATA RESULT MODEL BLUEPRINT
Path: `lib/core/models/sync_data_result.dart`

The `SyncDataResult` model is an **Isar Collection** used to track the history and status of synchronized production data packages. It serves as the data backing for the "Payload History" UI.

## 1. Collection Structure

| Field | Type | Description |
|---|---|---|
| `id` | `Id` | Primary Key (auto-increment). |
| `driverID` | `String?` | Unique ID of the operator (Indexed). |
| `status` | `String?` | Current state: `"pending"` or `"sent"` (Indexed). |
| `shift` | `String?` | Logical shift identifier: `"pagi"` or `"malam"`. |
| `shiftTime` | `int?` | Epoch of the start of the shift (Indexed). Used for grouping spots. |
| `startTime` | `int?` | Timestamp of the first spot in the package. |
| `endTime` | `int?` | Timestamp of the last spot in the package. |
| `syncTime` | `int?` | Timestamp when the payload was successfully sent. |
| `totalSpot` | `int?` | Count of WorkingSpots contained in this payload. |

---

## 2. Key Features

- **Reactivity**: The UI watches this collection via `watchSyncDataResults(driverId)` in `AppRepository`.
- **Relationship**: This model does NOT contain the actual spot data. It references a "Group" of spots filtered by `driverID` and `shiftTime` in the `WorkingSpot` collection.
- **Auto-Generation**: Pending records are generated by calling `AppRepository.generatePendingSyncData(driverID)`.

---

## 3. Data Integration Flow

1. **Detection**: On login or page load, the app checks for shifts that haven't been synced.
2. **Creation**: A `SyncDataResult` is created with status `"pending"`.
3. **Execution**: When "Kirim" is pressed, the presenter queries `WorkingSpot` based on the `shiftTime` stored in this model.
4. **Finalization**: Upon successful binary transmission, the `status` is updated to `"sent"` and `syncTime` is recorded.

---

> [!TIP]
> Always use `shiftTime` (base epoch of the shift day) for querying associated records to ensure data integrity across multiple workfiles in the same shift.
